4.22.2. IntroductionÔÉÅ

This document describes the functionality and configuration of the AUTOSAR BSW module Wdg.

Supported AUTOSAR Release

4.3.1

Supported Configuration Variants

Pre-Compile, Post-build & Link-Time

Vendor ID

WDG_VENDOR_ID (44)

WDG_MODULE_ID (102)

Supported Platform

AM263x

The WDG driver depends on some other modules, like DET for error handling, OSEK/AUTOSAR OS for interrupt handling and the Mcu module for system clock settings.

4.22.3. Functional OverviewÔÉÅ

4.22.3.1. WDG Driver ArchitectureÔÉÅ

The Wdg driver has architecture as shown in below figure from the AUTOSAR spec

The WDG driver provides services for handling the watch dog timer on the microcontroller.

The WDG driver performs two main tasks, which are as follows:

  1. Initialization of WDG

  2. Reset WDG expiration time (trigger)

4.22.3.2. InitializationÔÉÅ

The initialization covers the following steps

  1. Set selected watchdog timer expiration time

  2. Set selected trigger timeout

  3. Set selected window size (Digital Windowed Watchdog)

  4. Set selected mode (Fast or slow can be selected and can be switched runtime)

4.22.3.3. StatesÔÉÅ

  1. WDG_IDLE: This is indicating the module is initialized and not busy.

  2. WDG_BUSY: This is indicating the module is busy.

4.22.3.4. AssumptionsÔÉÅ

  1. The clock and PLL configuration is outside driver and should be configured for proper WDG operation.

  2. The Pin Muxing should be initialized before initializing the Wdg module.

4.22.3.5. LimitationsÔÉÅ

  1. Wdg_SetTriggerCondition(0): If ‚Äú0‚Äù is passed as the parameter for Wdg_SetTriggerCondition will result in an immediate stop of the watchdog triggering and an instantaneous watchdog reset of the ECU (irrespective of whether reaction is NMI or reset).

4.22.3.6. Design overviewÔÉÅ

Please refer SITARA MCU MCAL Architecture Document and MCAL: WDG Detailed Design Document provided as part of CSP.

4.22.4. Hardware FeaturesÔÉÅ

4.22.4.1. Hardware Features supportedÔÉÅ

  • Setting default mode and timeout during initialization.

  • Setting window size (Digital Windowed Watchdog).

  • Enable watchdog timer.

  • Service watchdog timer.

  • Mode can be changed at runtime

4.22.4.2. Not supported FeaturesÔÉÅ

  • Off mode not supported

4.22.8. Integration descriptionÔÉÅ

4.22.8.1. Dependent modulesÔÉÅ

4.22.8.1.1. MCUÔÉÅ

The module MCU powers up the microcontroller‚Äôs peripherals at startup time and initializes the PLL.

4.22.8.1.2. OSÔÉÅ

An operating system can be used for task scheduling, interrupt handling, global suspend and restore of interrupts and creating of the Interrupt Vector Table. The WDG module may use AUTOSAR OS to suspend and restore global interrupts.

4.22.8.1.3. SchMÔÉÅ

There are no exclusive areas.

4.22.8.1.4. Critical SectionsÔÉÅ

There are no critical sections.

4.22.8.1.5. DEMÔÉÅ

By default, production code related errors are reported to the DEM using the service DEM_ReportErrorStatus(). Note: Dem Event is enable only if $(Module_Name)DemEventParameterRefs is enabled.

4.22.8.1.6. DETÔÉÅ

By default, production code related errors are reported to the DET using the service Det_ReportError(). If development error reporting is enabled (i.e. pre-compile parameter WDG_DEV_ERROR_DETECT== STD_ON). The reported service IDs identify the services which are described in 3.4.

4.22.8.1.7. Callback NotificationÔÉÅ

The WDG driver does not provide any call back functions.

4.22.8.2. Multi-core supportÔÉÅ

Not supported

4.22.10. ExamplesÔÉÅ

4.22.10.1. OverviewÔÉÅ

  1. Initializes clock using Mcu_Init()

  2. Configure the interrupts for WDG using WDTApp_interruptConfig()

  3. Default Example program is configured in Non-Maskable Interrupt method (WDT will trigger interrupt when counter becomes zero).

  4. WDT window reaction can be changed to ‚Äúreset‚Äù through configuration (EB Tresos).

  5. Get Watchdog version using Wdg_GetVersionInfo()

  6. Calculate and define the triggerValue and maxServiceInterval

  7. In example, service interval is 25ms and service count is 100. So, every 25ms core will service the WDT for 100 times (25*100 = 2500 ms).

  8. Initialize the timer using Gpt_Init()

  9. Initialize the Watchdog using Wdg_Init()

  10. Start the timer using API Gpt_StartTimer()

  11. Set the trigger condition using Wdg_SetTriggerCondition() along with timeout value (Timeout is defined by subtracting triggerValue with maxServiceInterval)

  12. Then after it will raise an interrupt and interrupt will run (WDT_Isr() function). In ISR function core reset will occurs and start running program again automatically.

  13. On reset, Check the reason for the reset using GetResetReason(). If reset happened due to WDT then ‚ÄúTest Passed‚Äù

  14. In Reset window reaction mode, after counter reaches to zero POR(Power ON Reset) happens.

4.22.10.2. Hardware Software Setup and ToolsÔÉÅ

None

4.22.10.3. Steps to build and run exampleÔÉÅ

  1. WDG example application demonstrates the MCAL WDG driver features, which is referred from folder <MCAL_ROOT>/examples/WDG.

  2. This application can be built from the build folder by giving ‚Äúgmake ‚Äìs wdg_app PLATFORM=am263‚Äù command in command prompt.

  3. Once the build is completed we get an appimage file, which is flashed in am263x controller and executed.

  4. To Flash the appimage at the respective memory location as the SBL expects, path of the appimage file is provided in ‚Äúdefault_sbl_qspi‚Äù file in ‚Äú..<MCU_PLUS_SDK_INSTALLATION>\tools\boot\sbl_prebuilt\am263x-cc‚Äù

  5. Set the boot mode of board to UART mode.(UART boot mode: 1000)

  6. Open command prompt at the path ‚Äú..<MCU_PLUS_SDK_INSTALLATION>\tools\boot‚Äù and provide command as follows: ‚Äúpython uart_uniflash.py -p COM<x> ‚Äìcfg=sbl_prebuilt/am263x-cc/default_sbl_qspi.cfg‚Äù Here <x> depicts the COM PORT number where hardware is connected to system, which can be detected from device manager

  7. Once the appimage is flashed after the command is executed, change the boot mode of board to QSPI mode.(QSPI boot mode: 0010)

  8. Open the terminal to get the expected application log after doing hard reset on board.

  9. Verify that the application produces the correct results as expected shown below.
